# machine_learning — architecture & boundary contract

## Guiding principle

**Services SENSE and ACT; the ML layer THINKS.** A service manager is reduced to three verbs — **FETCH** (HTTP GET against Radarr/Sonarr/Trakt/Tautulli), **CACHE** (project + persist raw responses to global_cache / Parquet), **APPLY** (HTTP PUT/DELETE/POST that the ML layer told it to make). Every value judgment — *is this watchable, what quality does it earn, should it be deleted/downgraded/upgraded, is it protected, when does its grace expire, what's the next episode to fetch, how big will it be, which library does it belong in* — lives under `scripts/managers/machine_learning/` (the "brain").

The brain is organized **by concern, not by service**, because the same decision (watchability scoring, size estimation, keep-policy resolution, deletion ranking) is duplicated across Radarr and Sonarr today. Pulling it into one concern-keyed tree is the whole point: `score_movie` and `score_show` used to share `_affinity`, `QUALITY_PROFILE_THRESHOLDS`, `_DEVICE_RESOLUTION_CEILING`, `_KIDS_CERTS`, `_TRANSCODE_FRIENDLY_CODECS` via a cross-service import (`shows/scorer.py` imported from `movies/scorer.py`) — now resolved: both import from `scoring/_shared.py`, a single shared space neither engine reads through the other (ML Step 2.1); space-pressure delete tiering is copy-pasted between `radarr/quality/space_pressure.py` and `sonarr/cache/episode_files.py`; size estimation already funnels through `size_model.py`. One brain ends the drift.

## The service ⇄ ML boundary contract

The boundary is **plain data in both directions** — no service imports a scorer, no ML module makes an HTTP call. Three rules enforce it:

1. **Services produce raw cache + feature rows.** A service fetches from its API, projects/caches the raw response (Parquet/global_cache), and exposes a *read* method (`load(instance) → DataFrame`, `get_people(tmdb)`, affinity cache keys). It builds a `contracts.MovieFeatureRow` / `ShowFeatureRow` / `EpisodeFeatureRow` from a cache row via `features/`. **This is the only place a Parquet column name or Trakt JSON shape is known.**

2. **ML consumes features, emits plans as plain data.** Every brain entrypoint has the signature `plan(features: FeatureRow|Iterable[FeatureRow], context: Context, config) → Plan|list[Plan]`. A `Plan` is a frozen dataclass: e.g. `QualityPlan(file_id, target_profile_id, direction, reason, est_space_gb_signed)`, `DeletePlan(file_id, reason, reclaim_gb)`, `MonitorPlan(movie_id, monitored, reason)`, `GracePlan(file_id, marked_for_deletion, available_until)`, `AcquirePlan(series_id, season, episode, target_profile_id)`. **No `requests`, no `radarr_api`, no `global_cache` writes inside the brain** — the brain is pure and unit-testable with hand-built feature rows (mirrors how `coordinator._select_for_target` is already documented as "Pure — unit-testable without the manager graph").

3. **Services apply plans.** The thin adapter takes `list[Plan]` and executes: `for p in plans: radarr_api.PUT(..., qualityProfileId=p.target_profile_id)`. The ledger stamp (`planned_action`/`plan_reason`/`plan_reclaim_gb`) is written by `ledger/` given a plan, then persisted by the service. In `dry_run`, the service stamps the ledger and logs "would …" but issues no HTTP — exactly today's behavior, just with the decision and the I/O on opposite sides of the seam.

**Coordinator special case (already half-correct):** `build_delete_candidates` (service: filters Sonarr/Radarr-owned lifecycle rows, applies whole-file guards that need Sonarr metadata) stays in the service but emits `contracts.DeleteCandidate` dicts; `_select_for_target`/`_critic_sort` (pure ranking math) move to `space/coordinator_ranker.py`; `delete_selected_*` (HTTP DELETE + restore-set bookkeeping) stays in the service. The coordinator manager keeps Stage orchestration + free-space reads (FETCH), delegates ranking to `space/`, and calls the service `delete_selected_*` adapters (APPLY).

**Import direction (enforced one-way):** `services → contracts` and `services → ml.*plan() entrypoints`; `ml → contracts` only. A lint rule (`grep` gate in CI) forbids `import requests` / `radarr_api` / `sonarr_api` / `_make_request` anywhere under `machine_learning/`, and forbids `from scripts.managers.services.trakt.*.scorer import` (the old scorer location) anywhere outside `machine_learning/`.
