# machine_learning — incremental migration plan

## Incremental, test-gated migration (one concern at a time, system stays live)

**Golden rule:** every step *adds* the new ML module and *shims the old symbol to call it*, runs a parity gate, and only deletes the old body once green. Never move two concerns in one PR. The dry-run decision ledger (`plan_summary`) is the system-level parity oracle: a refactor is correct iff the `planned_action`/`plan_reclaim_gb` roll-up is byte-identical before and after on a captured cache snapshot.

### Step 0 — Scaffolding & guards (no behavior change)
- Create the package tree with `__init__.py` + docstrings only (every module a stub raising `NotImplementedError` or re-exporting nothing yet).
- Add `contracts/` dataclasses first — they are pure types, zero risk.
- Add CI guards: (a) `grep` gate forbidding `requests|_make_request|radarr_api|sonarr_api|trakt_api|tautulli_api` under `machine_learning/`; (b) gate forbidding new imports of `services.trakt.*.scorer` outside the brain.
- **Gate:** repo imports cleanly; full run + `plan_summary` snapshot captured as the golden baseline.
