# Comprehensive TV-franchise discovery

**Goal.** Auto-discover TV franchises (sibling/spin-off show groups) comprehensively ‚Äî both **same-named** (Law & Order(+SVU/OC), NCIS, CSI, Chicago, 9-1-1, Star Trek, Stargate) and **cross-named** (Grey's Anatomy‚ÜîStation 19‚ÜîPrivate Practice, Buffy‚ÜîAngel, Breaking Bad‚ÜîBetter Call Saul, Vampire Diaries‚ÜîOriginals‚ÜîLegacies, The Boys‚ÜîGen V) ‚Äî and feed them through **one seam** to playlist grouping, catch-up retention, AND acquisition. Movies need no work (already grouped by TMDB `collection_tmdb_id`).

Status: **Phase 0 + Phase 1 + Phase 2 (Wikidata edge + loader + floor) landed (2026-06-21); Phase 3 pending.**

## Locked decisions (2026-06-21)
- **Acquire unowned siblings:** discovered franchises participate in acquisition ‚Äî engaging one show backfills the rest start-first (via the universe acquire plan + the coordinator's show-add path), bounded by `max_per_run`. Not playlist/retention-only.
- **Cross-validated source:** the cross-named catalog is generated from **both** the Wikipedia category `Television series by franchise` (human-curated membership) **and** Wikidata `P2512` spin-off edges (structured, tvdb-ids inline), merged/cross-checked.
- **Exhaustive first ship:** the full generated catalog ships baked (hundreds of franchises); the owned-tvdb intersection self-scopes it per Household.

## Architecture ‚Äî two layers, one seam, tvdb-keyed
**Layer 1 ‚Äî same-stem clustering (runtime, owned inventory, zero-maintenance).** Cluster owned Sonarr series by a shared subtitle-stem (`X` / `X: Sub`) or a distinctive leading token (the Chicago class). Self-configuring per Household; no baked list. Covers the same-named long tail (~40-55% of groups).

**Layer 2 ‚Äî cross-named catalog (generated, baked, diff-reviewed).** A reviewable `data/tv_franchises.generated.json` of `{franchise_key: {timeline:false, shows:[tvdb‚Ä¶], items:[{media:'show', tvdb}‚Ä¶], titles:[‚Ä¶]}}`, generated by a standalone CLI from a **graph of TV series** whose edges UNION **three cross-validated sources**:
  1. **Wikipedia category** `Television series by franchise` ‚Üí `Category:<X> television series` co-membership.
  2. **Wikidata** `P2512` (spin-off) + shared-`P179` edges.
  3. **Wikipedia article infobox `related`/`spinoffs` field + "Spin-offs"/"Crossovers" section links** ‚Äî the STRONGEST cross-named signal: it surfaces Grey's‚ÜíStation 19/Private Practice and Buffy‚ÜíAngel, which sources 1-2 miss. (Verified live.)

Every candidate node is **filtered to an actual TV series** ‚Äî keep only links that resolve to a Wikidata item carrying a TVDB id (`P4835`); this drops the `related`-field noise (genres, people, places, file images) AND the `P179` over-capture (bare `P179` on NCIS returns 498 episode/season rows ‚Äî the filter is load-bearing). **Connected components** over the unioned edges = franchises; each node resolved to its tvdb id. Config-overlayable via a new `plex.playlists.tv_franchises` key (like `universe_lists`).

**The single seam.** A pure `tv_franchise_universes(owned_series_rows, catalog, *, cluster_same_stem=True)` emits **synthetic universe entries** `{key: {timeline:False, movies:[], shows:[tvdb‚Ä¶], items:[{media:'show', tvdb}‚Ä¶]}}` (namespaced keys `tvfran:‚Ä¶` so they never collide with film universes). `builder._universe_source` merges these into the returned `{"universes": ‚Ä¶}` **after** the mdblist universes. That one merge reaches **all three** consumers unchanged:
- playlist grouping (`build_universe_maps` / `tv_group_maps`),
- catch-up retention (`saga_member_sets` ‚Äî closes its curated-franchise blind spot),
- acquisition (`unified_universe_order` ‚Üí `universe_acquire_plan`).

**Critical cache rule (as built ‚Äî corrects the original sketch):** the three downstream consumers split ‚Äî playlist grouping reads the LIVE `builder._universe_source()`, but catch-up retention, hybrid acquisition AND the Sonarr prefetch walk read the `plex/playlists/universe_source` cache key DIRECTLY. So synthetic entries are written to the CACHE (not merged into a returned dict only), regenerated every run: `builder._refresh_synthetic_universes` strips all prior `tvfran:*` keys then re-adds the fresh set, preserving the mdblist universes + their TTL `fetched` ordinals verbatim (a `__tvfran__` bookkeeping marker only). Run-order: the playlist builder writes during the Plex phase (before retention/acquisition); the Sonarr prefetch walk runs earlier, so it sees the PRIOR run's set (a one-run lag, same as it already has for mdblist ‚Äî acceptable).

Entries emit **`timeline: True`** (NOT False): `unified_universe_order` SKIPS non-timeline universes, so the acquisition backfill would never see a franchise's gaps with `timeline:False`. They carry `movies:[]` + a debut-ordered `shows:[tvdb‚Ä¶]` + `items:[{media:'show', tvdb, rank}‚Ä¶]` (all three are load-bearing: `shows`‚Üí`build_universe_maps` grouping/order, `items`‚Üí`saga_member_sets` + `unified_universe_order`). `tv_group_maps` precedence: a real mdblist list still wins over a curated name, but a synthetic `tvfran:` cluster YIELDS to the hand-named curated franchise, so "One Chicago"/"Law & Order" keep their label in the playlist while still feeding retention + acquisition via the cache.

## Distribution ‚Äî split by media (decided)
The two media live in DIFFERENT homes; the volume cuts opposite ways.

**MOVIE universes ‚Üí the canonical `Glidearr` mdblist account.** ~15-20 IMDb timeline lists imported
once (auto-syncing, public), fits the **Plus** tier (20 external lists). Deployments read them by id
with their OWN mdblist key (public lists are cross-account readable; the account's API quota is only
for the curator publishing). Glidearr OWNS them, so a stranger deleting a community list can't break
a default. The `UNIVERSE_LISTS` refs migrate to `glidearr/<slug>`.

**TV franchises ‚Üí a generated JSON catalog FILE, NOT mdblist.** One franchise = one mdblist list ‚Üí
hundreds of franchises would blow past mdblist's caps (Plus 40, max 300) for ‚Ç¨150-200/yr to hold what
fits in one ~50 KB file ‚Äî AND a franchise has no single syncable source URL (membership comes from the
Wikipedia/Wikidata graph), so mdblist's auto-sync buys nothing. So:
- **Baked floor** (`data/tv_franchises.json`, in the package): a SMALL high-confidence set (Chicago,
  Arrowverse, Grey's, 9-1-1, ‚Ä¶) ‚Äî keyless/offline, diff-reviewed.
- **Full catalog**: the exhaustive generated set (basic + non-basic) ‚Äî same JSON shape ‚Äî overlays the
  floor. **JSON not parquet** (binary parquet hides a bad Wikidata edit in diff; JSON is PR-reviewable
  ‚Äî the vandalism guard).
- **Update without a code release ‚Äî two levers, neither is mdblist:** (1) config overlay
  `plex.playlists.tv_franchises` in the appdata `config.json` (one line, no rebuild); (2) optional
  fetch of the latest `tv_franchises.json` from ONE canonical Glidearr URL (raw GitHub / tiny
  endpoint) ‚Äî centralized, one file, free, no list cap.

Both the baked floor and the config/URL overlay normalize to the same `tv_franchise_universes()`
synthetic-entry shape, so the seam is identical regardless of source.

## Risks / guards
- **P179 over-capture** ‚Üí the `P4835` TVDB-series filter is mandatory + unit-tested with a stubbed SPARQL fixture.
- **Same-stem false-merge** (The Office US/UK, Shameless US/UK; generic `American */The *`) ‚Üí stoplist + DENY set + min-cluster=2.
- **Cache hazard** ‚Üí synthetic entries merged into the RETURNED dict only, regenerated each run.
- **Wikidata vandalism** ‚Üí mandatory diff-review of the generated JSON before commit; never auto-commit.
- **Acquisition blast radius** ‚Üí a franchise with many unowned siblings (NCIS) backfills a lot; bounded by `max_per_run` + start-first ordering. Sibling-show acquisition rides the universe coordinator's show-add path (still to build).
- **Member order** must be defensible debut/air order (the retention watchlist-prefix scoping relies on rank).
